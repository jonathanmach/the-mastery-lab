## 1. Purpose & Scope

This prototype demonstrates a **clinical data search platform** built on open FHIR standards. It:

- Ingests synthetic patient data (generated by [Synthea](https://github.com/synthetichealth/synthea)) in FHIR R4 Bundle format
- Persists resources in a HAPI FHIR R4 server (the system of record)
- Builds a denormalized search index in OpenSearch for fast faceted search
- Exposes a FastAPI Backend-for-Frontend (BFF) layer
- Renders a Vue 3 SPA with a patient search and detail view

The system is **not production-ready**. Security (OpenSearch auth, TLS, RBAC) is disabled for local development simplicity.

## 4. Infrastructure: Docker Compose

All infrastructure services are defined in `docker-compose.yml`.

### Services

#### HAPI FHIR

```yaml
image: hapiproject/hapi:latest
ports: "8081:8080"
```

- FHIR version: **R4**
- Storage: H2 embedded file database at `/data/hapi/hapi` (volume `hapi_data`)
- Key config:
  - `hapi.fhir.allow_multiple_delete=true`
  - `hapi.fhir.reuse_cached_search_results_millis=0` (disable caching)
- JVM memory: `-Xms1g -Xmx2g`
- Health check: polls `/fhir/metadata` every 15 s
- Public URL (from host): `http://localhost:8081/fhir`

HAPI provides a standard FHIR REST API. Resources are stored as-is and can be retrieved, searched, or updated via HTTP.

#### OpenSearch

```yaml
image: opensearchproject/opensearch:2.11.0
ports: "9200:9200"
```

- Single-node, no authentication (`DISABLE_SECURITY_PLUGIN=true`)
- JVM memory: `-Xms512m -Xmx512m`
- Persistent data via volume `opensearch_data`
- Health check: `curl -f http://localhost:9200` every 10 s

#### OpenSearch Dashboards

```yaml
image: opensearchproject/opensearch-dashboards:2.11.0
ports: "5601:5601"
```

Useful during development for inspecting the `patients` index directly. Not used by the application.

### Starting Infrastructure

```bash
docker compose up -d
# Wait for HAPI health check to pass (30–60 s on first start)
```

## 6. Ingestion Pipeline

### Overview

The ingestion script (`scripts/ingest_bundles.py`) is a one-shot CLI tool. It reads FHIR R4 JSON bundles from disk, validates them, uploads to HAPI FHIR, and builds the OpenSearch index.

```bash
uv run python scripts/ingest_bundles.py [--dir ./data/synthea_bundles]
```

### Step-by-Step Flow

```
1. Initialize:
   - Load Settings (from .env)
   - Create FHIRClient
   - Create SearchProjector (creates OpenSearch index if missing)
   - Create ImportTracker (creates SQLite tables if missing)

2. Load infrastructure bundles:
   - Files named *hospital* or *practitioner* are posted first
   - These create Organization & Location resources in HAPI
   - Posted as-is (transaction bundles with POST verbs)

3. Pre-create Practitioner stubs:
   - Scan all patient bundles for embedded Practitioner NPI identifiers
   - Create minimal Practitioner resources with deterministic UUIDs (hash of NPI)
   - Upload in batches of 200 via transaction bundle
   - Prevents reference resolution failures when patient bundles reference practitioners

4. For each patient bundle file:
   a. source_reader.read_bundle_from_file(path)  → raw dict
   b. bundle_parser.parse_bundle(raw)            → ParseResult (validates each resource)
   c. Record validation errors in ImportTracker (status=failed for invalid resources)
   d. bundle_parser.prepare_bundle_for_upsert()  → rewritten bundle (POST → PUT with IDs)
   e. fhir_client.post_transaction_bundle()      → POST to HAPI FHIR
   f. search_projector.project_patient()         → fetch from HAPI, index to OpenSearch
   g. ImportTracker.mark_success() or mark_failed()

5. Log final summary:
   INFO  ingest  Ingestion complete — success: N  failed: M  projected patients: P
```

### Bundle Parser (`src/app/ingestion/bundle_parser.py`)

**`parse_bundle(raw_dict) → ParseResult`**

- Validates the envelope: must have `resourceType=Bundle` and `entry` list
- For each entry, extracts the resource type and attempts to parse using the `fhir.resources` R4B library
- Supported types: `Patient`, `Encounter`, `Condition`, `Observation`, `MedicationRequest`
- Returns a `ParseResult` containing:
  - `resources`: list of successfully parsed `ResourceResult` objects
  - `errors`: list of validation error messages

Validation failures per-resource are recorded but do not abort the bundle — other resources in the bundle continue to be processed.

**`prepare_bundle_for_upsert(raw_dict) → dict`**

- Rewrites request entries from `POST` to `PUT` with explicit resource IDs
- Synthea generates UUID-based resource IDs (e.g., `urn:uuid:abc123`)
- The rewritten bundle uses `PUT [ResourceType]/[id]` format
- This ensures HAPI stores resources with the Synthea-assigned UUIDs rather than server-assigned IDs, making subsequent lookups deterministic
- HAPI still resolves internal `urn:uuid:` cross-references correctly

### FHIR Client (`src/app/ingestion/fhir_client.py`)

Thin wrapper around HAPI's FHIR REST API using `httpx.Client`.

**Write methods:**
- `upsert_resource(resource_type, logical_id, resource_dict)` — `PUT /fhir/{type}/{id}`
- `post_transaction_bundle(bundle_dict)` — `POST /fhir` (FHIR transaction)

**Read methods:**
- `get_resource(resource_type, logical_id)` — `GET /fhir/{type}/{id}`
- `search(resource_type, params)` — `GET /fhir/{type}?{params}` → Bundle
- `get_patient_resources(patient_id, resource_type)` — paginated search for all resources belonging to a patient

Timeouts: 10 s connect, 120 s read.

### Import Tracker (`src/app/ingestion/import_tracker.py`)

SQLite-backed audit trail using SQLAlchemy. Tracks every resource processed during ingestion.

```
ImportRecord:
  id               UUID primary key
  source_file      TEXT (indexed)   — original bundle filename
  bundle_id        TEXT             — FHIR Bundle.id
  resource_type    TEXT             — Patient, Encounter, etc.
  logical_id       TEXT (indexed)   — FHIR resource ID
  import_status    TEXT             — pending | success | failed | skipped
  validation_errors TEXT            — JSON array of error strings
  import_timestamp DATETIME
```

## 7. Search Projection Layer

### Responsibility

`SearchProjector` (`src/app/projection/search_projector.py`) owns two concerns:

1. **Indexing** — translating a patient's FHIR resources into a flat OpenSearch document
2. **Querying** — executing search and facet queries against that index

### `project_patient(patient_id, fhir_client)`

Called once per patient after successful ingestion. Fetches all relevant resources from HAPI and builds the index document:

```python
patient   = fhir_client.get_resource("Patient", patient_id)
encounters = fhir_client.get_patient_resources(patient_id, "Encounter")
conditions = fhir_client.get_patient_resources(patient_id, "Condition")
observations = fhir_client.get_patient_resources(patient_id, "Observation")
medications = fhir_client.get_patient_resources(patient_id, "MedicationRequest")

doc = self._build_doc(patient, encounters, conditions, observations, medications)
self.client.index(index=INDEX, id=patient_id, body=doc)
```

### `_build_doc(...)` — Denormalization Logic

The document schema is designed for both full-text search and faceted aggregation:

| Field | Type | Description |
|---|---|---|
| `patient_id` | keyword | FHIR Patient resource ID |
| `name` | text + keyword | Full name (for text search and sorting) |
| `family_name` | keyword | Family name (for sorting) |
| `given_name` | keyword | Given name |
| `gender` | keyword | male / female / other / unknown |
| `birth_date` | date | ISO-8601 date |
| `age_band` | keyword | <18 / 18-34 / 35-49 / 50-64 / 65+ |
| `conditions` | text | Human-readable condition display names |
| `condition_codes` | keyword[] | SNOMED CT codes |
| `condition_pairs` | keyword[] | `"code\|display"` — used for facet aggregations |
| `medications` | text | Human-readable medication names |
| `medication_codes` | keyword[] | RxNorm codes |
| `medication_pairs` | keyword[] | `"code\|display"` pairs |
| `observation_codes` | keyword[] | LOINC codes |
| `observation_pairs` | keyword[] | `"code\|display"` pairs |
| `obs_values` | object (dynamic) | Numeric values keyed by LOINC code (e.g., `"8302-2": 174.5`) |
| `latest_observations` | object (disabled) | Full obs data, not indexed |
| `last_encounter_date` | date | Most recent encounter |
| `has_active_medication` | boolean | Any MedicationRequest with `status=active` |
| `has_recent_encounter` | boolean | Encounter within past 365 days |
| `validation_status` | keyword | insufficient / partial / usable |

**Validation status logic:**

| Status | Meaning |
|---|---|
| `usable` | Has Patient + Encounter + (Condition or Observation) |
| `partial` | Has Patient + Encounter but no clinical data |
| `insufficient` | Missing Patient or Encounter |

Only `usable` and `partial` patients appear in search results by default.

**Code + Display pair format:**

Facet aggregations need both a machine-readable code and a human-readable label. Rather than using a nested object (which requires `nested` queries), these are concatenated as `"code|display"` strings in keyword arrays. When building facet buckets, the BFF splits on `|` to recover both values.

### `search(q, gender, condition, medication, age_band, recent_encounter, observation, obs_min, obs_max, page)`

Builds an OpenSearch `bool` query:

- `must`: term queries for each active filter
- `filter`: range query if `obs_min`/`obs_max` provided for an observation code
- `should`: multi-match on `name`, `family_name`, `given_name` if `q` is provided
- Pagination: `from = (page - 1) * 20`, `size = 20`
- Sort: `family_name.keyword` ascending

### `facets(gender, condition, medication, age_band, observation)`

Runs a `size=0` aggregation query. Returns term counts for:
- `gender`
- `age_band`
- `condition_pairs` (top 50)
- `medication_pairs` (top 50)
- `observation_pairs` (top 50)
- `has_recent_encounter`

Facets are computed against the same filter context as the current search, so they reflect "what is possible given current filters" rather than global totals.

### Views

#### `PatientListView.vue`

Layout: two-column flex.

- **Left column**: `<FacetPanel>` — reads `store.facets`, emits filter changes to the store
- **Right column**:
  - Search bar bound to `store.filters.q` via `setFilter('q', value)`
  - Result count + loading state
  - `v-for` grid of `<PatientCard>` components
  - Pagination: prev/next buttons calling `store.prevPage()` / `store.nextPage()`

#### `PatientDetailView.vue`

Fetches data on mount via `getPatientSummary()` and `getPatientTimeline()`.

Tabs:
1. **Summary** — conditions table, medications table, observations grid (latest value per code)
2. **Timeline** — events sorted newest-first, grouped by year. Event types: Encounter (blue), Condition (red), Observation (green), MedicationRequest (orange)
3. **Raw FHIR** — calls `getPatientResources()` and renders JSON in `<pre>` blocks per resource type

- Receives `facets` and `activeFilters` as props
- Emits `filter-change(key, value)` events to parent
- Each section (Gender, Age Band, Diagnosis, Medications, Observations) is collapsible
- Active filter count displayed as badge on section header
- Observations section includes numeric range inputs (`obs_min`, `obs_max`)
- "Clear all" button resets everything

```
           ▼                          ▼
┌──────────────────┐       ┌──────────────────────────┐

│  OpenSearch      │
│  :9200           │
│  index: patients │

           │
           ▼
┌──────────────────┐   ──────────────────────────────────
│  FastAPI BFF     │   GET /patients/search?q=...&gender=
│  :8000           │   GET /patients/facets
│                  │   GET /patients/{id}/summary
│  routes.py       │   GET /patients/{id}/timeline

           │  (Vite proxy: /patients → :8000)
           ▼
┌──────────────────────────────────────────────────────┐
│  Vue 3 SPA  :5173                                    │
│                                                      │
│  PatientListView        PatientDetailView            │
│   FacetPanel             Summary tab                 │
│   PatientCard            Timeline tab                │
│   search input           Raw FHIR tab               │
└──────────────────────────────────────────────────────┘
```

### Data Stages in Detail

**Stage 1 — Ingest**
Raw Synthea bundles are read from `data/synthea_bundles/*.json`, validated resource-by-resource, then posted as FHIR transactions to HAPI. Practitioner stubs are created first to satisfy reference integrity. Infrastructure bundles (hospitals) are loaded before patient bundles.

**Stage 2 — Project**
Immediately after each successful FHIR upload, `project_patient` fetches the full patient record back from HAPI (ensuring we project exactly what was stored) and builds a denormalized OpenSearch document. This fan-out from HAPI keeps the index consistent with the canonical record.

**Stage 3 — Search & Facet**
The frontend sends a single search request that hits FastAPI, which translates filters into an OpenSearch `bool` query and a parallel aggregation query. Results and facets are returned in a single API response cycle.

**Stage 4 — Detail**
Clicking a patient card navigates to the detail view. FastAPI fetches raw resources from HAPI (not from OpenSearch), ensuring the detail view always reflects the canonical FHIR record. The timeline and summary are assembled server-side from those raw resources.

## 11. Database Schema

SQLite file at `import_tracking.db` (created automatically).

```sql
CREATE TABLE import_records (
    id               TEXT PRIMARY KEY,          -- UUID
    source_file      TEXT NOT NULL,             -- bundle filename
    bundle_id        TEXT,                      -- FHIR Bundle.id
    resource_type    TEXT NOT NULL,             -- Patient | Condition | ...
    logical_id       TEXT,                      -- FHIR resource ID
    import_status    TEXT NOT NULL,             -- pending | success | failed | skipped
    validation_errors TEXT,                     -- JSON array of error strings
    import_timestamp DATETIME
);

CREATE INDEX ix_import_records_source_file ON import_records(source_file);
CREATE INDEX ix_import_records_logical_id  ON import_records(logical_id);
```

The tracker supports re-running ingestion: previously successful records can be detected and skipped (though in the current prototype, re-running simply overwrites records).

```json
      "gender":              { "type": "keyword" },
      "birth_date":          { "type": "date" },
      "age_band":            { "type": "keyword" },
      "conditions":          { "type": "text" },
      "condition_codes":     { "type": "keyword" },
      "condition_pairs":     { "type": "keyword" },
      "medications":         { "type": "text" },
      "medication_codes":    { "type": "keyword" },
      "medication_pairs":    { "type": "keyword" },
      "observation_codes":   { "type": "keyword" },
      "observation_pairs":   { "type": "keyword" },
      "obs_values":          { "type": "object", "dynamic": true },
      "latest_observations": { "type": "object", "enabled": false },
      "last_encounter_date": { "type": "date" },
      "has_active_medication":  { "type": "boolean" },
      "has_recent_encounter":   { "type": "boolean" },
      "validation_status":   { "type": "keyword" }
    }
  }
}
```

**Notes:**
- `obs_values` is a dynamic object. Numeric observation values are stored as `obs_values["8302-2"] = 174.5`. OpenSearch auto-maps each LOINC code as a `float` field inside the object.
- `latest_observations` stores the full observation JSON but is `enabled: false` — it is stored but never indexed or searchable. Used only for retrieval.
- `name` has both a `text` mapping (for full-text search) and a `keyword` subfield (for exact match and sorting).

## 14. Key Design Decisions

### Backend-for-Frontend (BFF) Pattern

FastAPI does not expose HAPI FHIR or OpenSearch directly to the client. Instead, it provides purpose-built endpoints shaped for the UI. This isolates the frontend from FHIR query complexity and enables response aggregation (e.g., fetching 5 resource types for a patient detail in one API call).

### Dual-Store Architecture (HAPI + OpenSearch)

HAPI FHIR is the **system of record**. OpenSearch is a **derived read model**.

- HAPI provides compliance, validation, and correct FHIR semantics
- OpenSearch provides faceting and full-text search that HAPI's query language cannot efficiently express
- The projection step (`project_patient`) keeps them in sync
- The trade-off: complexity (two stores to manage) vs. query performance

If OpenSearch were corrupted, re-running `ingest_bundles.py` would rebuild it from HAPI.

### Denormalization via "Pair" Fields

FHIR codes (SNOMED CT, LOINC, RxNorm) are machine-readable but not human-readable. The UI needs to show display names in filter chips. Rather than storing codes and displays in a `nested` object (which requires nested queries), we concatenate them as `"code|display"` keyword strings. A single `terms` aggregation returns both values, and the BFF splits on `|` to separate them.

### Preserving Synthea UUIDs

Synthea generates consistent, deterministic UUIDs for all resources within a bundle. The ingestion pipeline rewrites bundle entries to use `PUT /Resource/{id}` rather than `POST /Resource`, preserving these IDs. This means that:
- Re-ingesting the same bundle is idempotent (HAPI updates rather than creates duplicates)
- Lookups by ID in `project_patient` work predictably without a secondary ID resolution step

### Validation Status

Not all Synthea bundles are complete. The `validation_status` field (`insufficient` / `partial` / `usable`) gives the search UI a way to filter out low-quality records. Only patients with at least a Patient resource and an Encounter are shown in search by default.

### Vite Proxy

The frontend development server proxies all `/patients` requests to the FastAPI backend. This avoids CORS configuration complexity and mirrors a production deployment pattern where a reverse proxy (nginx, Caddy, AWS ALB) would route `/patients` to the backend.

## 15. Reproducing from Scratch

### Prerequisites

- Docker Desktop (or Docker + Docker Compose)
- Python 3.11+ with [uv](https://github.com/astral-sh/uv) installed
- Node.js 18+ and npm

### Step 1: Clone & Enter the Prototype Directory

```bash
cd prototypes/fhir-platform
```

### Step 2: Start Infrastructure

```bash
docker compose up -d
```

Wait ~60 seconds for HAPI FHIR to initialize (watch: `docker compose logs -f hapi`). HAPI is ready when you see `Started Application` in the logs or when `http://localhost:8081/fhir/metadata` returns a CapabilityStatement.

### Step 3: Install Python Dependencies

```bash
uv sync
```

### Step 4: Configure Environment

```bash
cp .env.example .env
# Edit .env if your service ports differ from defaults
```

### Step 5: Download Synthetic Patient Data

Download [Synthea](https://github.com/synthetichealth/synthea) and generate bundles, or download the [pre-generated 1000-patient dataset](https://synthea.mitre.org/downloads).

Place `.json` bundle files in `data/synthea_bundles/`.

For a quick test, 10–20 patient bundles are sufficient.

### Step 6: Run Ingestion

```bash
uv run python scripts/ingest_bundles.py
```

Expected output:
```
INFO  ingest  Processing bundle_abc.json
INFO  ingest  Projection succeeded for patient <uuid>
...
INFO  ingest  Ingestion complete — success: 50  failed: 0  projected patients: 50
```

### Step 7: Start the Backend API

```bash
uv run fastapi dev src/app/api/main.py
```

Verify: `http://localhost:8000/health` → `{"status": "ok"}`
API docs: `http://localhost:8000/docs`

### Step 8: Start the Frontend

```bash
cd frontend
npm install
npm run dev
```

Open `http://localhost:5173` in your browser.

### Service URLs Summary

| Service | URL | Purpose |
|---|---|---|
| Vue SPA | http://localhost:5173 | Main application |
| FastAPI | http://localhost:8000 | Backend API (+ /docs) |
| HAPI FHIR | http://localhost:8081/fhir | FHIR server |
| OpenSearch | http://localhost:9200 | Search engine |
| OS Dashboards | http://localhost:5601 | Index inspection (dev) |

### Rebuilding the OpenSearch Index

If you want to re-index without re-uploading to HAPI:

```bash
# Delete existing index
curl -X DELETE http://localhost:9200/patients

# Re-run ingestion (HAPI already has the data; projection will re-create the index)
uv run python scripts/ingest_bundles.py
```

### Common Issues

**HAPI not ready during ingestion:** Wait longer after `docker compose up`. Check `docker compose logs hapi`.

**OpenSearch index already exists:** Safe to re-run ingestion — the projector will update (not recreate) documents.

**Frontend shows "Network error":** Ensure FastAPI is running on port 8000 and the Vite proxy is active.

**`urn:uuid:` reference errors in HAPI:** Ensure practitioner stubs are created first (the ingestion script does this automatically).
